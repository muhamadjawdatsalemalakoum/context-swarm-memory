# Reproducing the CSM Benchmark

## What this guide does

This guide takes you from a fresh clone of the repository to a source-level validation of the benchmark harness without any LLM API calls. The source code, tests, synthetic corpus, queries, plotting/report scripts, and small canonical v0.2 result rows are versioned in git. Large local caches, embeddings, sidecar indexes, and ad-hoc run folders remain ignored.

The harness is cache-first: every prompt sent to Ollama during a bench run is content-hashed and the responses are stored under `data/eval/runs/<runId>/`. The committed canonical result rows let you verify the public accuracy/citation claims without an LLM or GPU; a full fresh bench is included for anyone who wants to regenerate the response cache on their own 4090.

## Prerequisites

- **Node.js ≥ 20** and **npm** (Node 20 is the engine declared in `package.json`).
- **PowerShell or bash.** Examples below use PowerShell syntax (`$env:VAR="value"`); bash users substitute `export VAR=value`.
- **No GPU required for source validation.** `npm test` and `npm run bench:smoke` run on CPU with no API key.
- **No GPU required for replay once artifacts are present.** `npm run bench:replay -- <runId>` only reads the saved run's `results.jsonl`.
- **For a fresh bench run only:** an **NVIDIA RTX 4090** (or equivalent 24 GB VRAM card) and a local **Ollama** install. The published numbers were gathered on a single 4090 — other hardware will produce different latency and may have different throughput characteristics.

## 1. Clone, install, and test

```powershell
git clone <repo-url> context-swarm-memory
cd context-swarm-memory
npm install
npm test
```

`npm test` runs the full vitest suite against `MockProvider`. It passes with **no API key and no GPU**. If this is red, stop here — nothing downstream will work.

## 2. Smoke run against MockProvider

```powershell
npm run bench:smoke
```

This invokes `csm bench run --trials 1 --corpus-sizes 100K --model-contexts 8K --queries q01,q11,q28`. It exercises the full bench pipeline end-to-end (corpus sampling, baseline routing, scorer, summarizer, output writer) against the real 9M-token synthetic corpus, but every LLM call goes through the deterministic mock. Expected wall clock: **a few seconds**. Numbers in the resulting `summary.json` are placeholders — this step only validates plumbing.

Outputs land under `data/eval/runs/run-<timestamp>/`:

- `results.jsonl` — one row per cell trial
- `summary.json` — aggregated cells
- (no `plots/` yet — that's generated by `bench:report`)

## 3. Verify the published benchmark evidence

The headline numbers in the README come from the canonical v0.2 rows committed under `data/eval/runs/`. First run the narrow verifier:

```powershell
npm run verify:published
```

This hashes the committed configs/result rows and recomputes the published counts, citation F1 values, and paired McNemar checks directly from `results.jsonl`. **No LLM calls, no GPU, no API key.**

You can also replay any published run summary from its result rows:

```powershell
npm run bench:replay -- <runId>
```

For example:

```powershell
npm run bench:replay -- v020-30q-embedfloor
npm run bench:replay -- lightrag-30q
npm run bench:replay -- scaling-rq1
npm run bench:replay -- scaling-1m
```

To regenerate the combined SOTA headline table from committed rows:

```powershell
npm run bench:sota:headline
```

To regenerate the scaling-SOTA slope report:

```powershell
npm run bench:sota:scaling
```

This recomputes the aggregated `summary.json` from the cached `results.jsonl` for that run. Expected wall clock: **under 5 minutes** end-to-end including disk I/O for the full 9M corpus sweep.

If the run directory is missing, replay will fail. Verify with:

```powershell
ls data/eval/runs/<runId>
```

You should see `results.jsonl` and the run config. For the public v0.2 claims, those files are tracked in git; release artifacts are only needed for larger caches or sidecar indexes that are intentionally not committed.

## 4. Fresh bench on your own 4090

Only do this if you want to regenerate the cache from scratch. It requires Ollama and a 4090-class GPU.

### 4a. Ollama setup

Install Ollama from <https://ollama.com> and confirm `ollama serve` is running (it usually starts as a service on install).

Set the five tuning env vars before launching the bench (PowerShell):

```powershell
$env:OLLAMA_FLASH_ATTENTION = "1"
$env:OLLAMA_KV_CACHE_TYPE   = "q8_0"
$env:OLLAMA_CONTEXT_LENGTH  = "8192"   # NOT OLLAMA_NUM_CTX (Ollama ignores that). Caps the context so
                                       # gemma4:31b stays fully on GPU; otherwise Ollama defaults to a
                                       # vram-based 32768 and the model spills to CPU (~15x slower).
$env:OLLAMA_KEEP_ALIVE      = "10m"
$env:OLLAMA_NUM_PARALLEL    = "1"
```

Bash equivalent:

```bash
export OLLAMA_FLASH_ATTENTION=1
export OLLAMA_KV_CACHE_TYPE=q8_0
export OLLAMA_CONTEXT_LENGTH=8192   # NOT OLLAMA_NUM_CTX (ignored); keeps gemma4:31b fully on GPU
export OLLAMA_KEEP_ALIVE=10m
export OLLAMA_NUM_PARALLEL=1
```

Optional thermal safety cap (cuts the 4090 from its 450 W default to 350 W; reduces throttling on long sweeps with marginal speed loss):

```powershell
nvidia-smi -pl 350
```

### 4b. Pull the models

```powershell
ollama pull gemma4:31b
ollama pull gemma4:e4b
```

`gemma4:31b` is ~17 GB on disk (Q4_K_M); `gemma4:e4b` is the smaller probe-stage model.

### 4c. Point CSM at Ollama

```powershell
$env:CSM_PROVIDER     = "ollama"
$env:CSM_PROBE_MODEL  = "gemma4:e4b"
$env:CSM_RECALL_MODEL = "gemma4:31b"
$env:CSM_SYNTH_MODEL  = "gemma4:31b"
```

Sanity-check the connection:

```powershell
npm run csm -- provider info
npm run csm -- provider ping --model gemma4:e4b
```

### 4d. Run the full sweep

```powershell
npm run bench:full
```

This is `csm bench run` with the default sweep matrix (corpus sizes × model contexts × four baseline systems × trials). Expected wall clock: **~30–60 minutes** on a 4090 against the 9M-token corpus. The runtime is dominated by the longest corpus × largest model-context cells; smaller cells finish in seconds. Filler tier expansion to 100M+ tokens significantly extends runtime — see Troubleshooting.

The run prints its progress to stderr and its `runId` at the end. Note that `runId` — you'll need it for the next step.

## 5. Render the plots

```powershell
npm run bench:report -- <runId>
```

This reads `data/eval/runs/<runId>/summary.json` and emits Vega-Lite specs under `data/eval/runs/<runId>/plots/` plus a `report.md` summary table. Each plot is a JSON file (e.g. `graph-A.vl.json`).

To render the specs to SVG or PNG, you have two options:

- **Online (no install):** open <https://vega.github.io/editor/>, paste the contents of a `.vl.json` file, and export from the menu.
- **Local CLI:** install the `vega-lite` package globally (`npm install -g vega-lite vega-cli`) and run `vl2svg plot.vl.json plot.svg` or `vl2png plot.vl.json plot.png`.

Optional flags pick which cell of the sweep is the "headline":

```powershell
npm run bench:report -- <runId> --headline-ctx 8K --headline-corpus 1M
```

## 6. SOTA sidecar runs

The SOTA sidecars are optional and slow. Start the sidecars as described in
`services/README.md`, then run:

```powershell
npm run bench:sota:smoke
npm run bench:sota:sidecars
```

`bench:sota:smoke` checks the CSM-vs-LightRAG path on three queries.
`bench:sota:sidecars` runs the currently wired SOTA adapters:
LightRAG, Mem0, and HippoRAG. If Mem0 or HippoRAG fails to install, index, or
query cleanly, preserve the failure log and mark it as blocked. Do not count a
sidecar failure as a CSM win.

## 7. Troubleshooting

**`Cannot find module @huggingface/transformers`** — run `npm install` first. This dependency powers the local embedding baselines and is installed from `package-lock.json`.

**`ECONNREFUSED localhost:11434`** — Ollama isn't running. Start it with `ollama serve` (or restart the Ollama desktop app). The default base URL the provider uses is `http://localhost:11434/v1`.

**Bench cell timeouts** — three knobs, in order of likelihood:

1. Reduce `--trials` (e.g. `--trials 1` for a quick pass).
2. Increase `OLLAMA_KEEP_ALIVE` (e.g. `30m`) so the model isn't reloaded between sequential calls.
3. Check GPU temperature with `nvidia-smi`. If you're thermally throttling, apply the `nvidia-smi -pl 350` cap or improve case airflow.

**Sweep crashes at 1B corpus size** — filler tiers beyond tier-3 are not shipped in this release; the corpus expander tops out at ~9M tokens. Constrain the sweep with `--corpus-sizes`:

```powershell
npm run bench:full -- --corpus-sizes 100K,1M,9M
```

This keeps you inside the available filler range. Larger sizes will become available in a later release once tier-4/5 filler is generated.
